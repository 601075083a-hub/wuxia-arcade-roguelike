# 战斗 HUD 规格与资源规范

> 状态：V1 规格先行  
> 目标平台：微信小游戏竖屏 Canvas 2D  
> 设计基准：390 x 844 逻辑坐标  
> 当前阶段：完整 Demo 的战斗界面垂直切片

## 1. HUD 目标

战斗 HUD 的首要目标不是展示所有系统，而是让玩家在 3 分钟内理解：

```text
选择碎片 -> 组成序列 -> 判断招式 -> 查看效果 -> 点击出招
```

首版 HUD 必须服务三个判断：

- 我现在有哪些手牌碎片。
- 我已经组成了什么序列，能不能出招。
- 出招会对敌人造成什么效果，敌人下一步要做什么。

首版不追求完整商业化 UI，不展示复杂天赋、战斗日志、长说明和完整状态详情。

## 2. 信息层级

### 2.1 必须常驻

这些信息在战斗中始终可见，不允许被角色、特效、微信胶囊或手势区遮挡。

| 优先级 | 信息 | 作用 |
| --- | --- | --- |
| P0 | 敌人名称、血量、意图 | 玩家判断本回合攻防目标 |
| P0 | 玩家血量、护甲 | 玩家判断生存压力 |
| P0 | 手牌碎片 | 玩家做搓招决策 |
| P0 | 当前搓招序列/招式名 | 玩家理解输入结果 |
| P0 | 出招按钮、回合按钮 | 玩家完成回合操作 |
| P1 | 出招次数 | 玩家判断资源限制 |
| P1 | 当前预计伤害/效果 | 玩家判断是否值得出招 |
| P2 | 内功/外功名称 | 只需要提醒构建存在，不抢主操作区 |

### 2.2 上下文显示

| 信息 | 显示时机 |
| --- | --- |
| 普通拳/普通腿提示 | 选择序列含拳或腿但未命中正式招式时 |
| 不可出招提示 | 只选择方向碎片时 |
| 已选碎片可撤回提示 | 选择至少 1 张牌后 |
| 内功触发提示 | 内功实际改变了出招次数、抽牌、弃牌或伤害时 |
| 敌人护盾/中毒/易伤 | 数值大于 0 时显示 |

## 3. 布局规格

所有坐标以 390 x 844 逻辑坐标为准。

### 3.1 安全区

| 区域 | 规则 |
| --- | --- |
| 顶部状态栏 | y < 32 不放关键 UI |
| 微信右上胶囊 | x >= 292 且 y <= 96 不放关键 UI |
| 底部手势区 | y >= 818 不放关键按钮和关键文字 |
| 横向边距 | 关键按钮和文本距离屏幕左右边缘至少 12px |

### 3.2 战斗页区域

| 区域 | 坐标建议 | 内容 | 设计要求 |
| --- | --- | --- | --- |
| 敌人 HUD | x=18, y=36, w=252, h=64 | 敌名、血条、血量、意图、护盾 | 避开微信胶囊，只占左上 |
| 敌人表现区 | x=70, y=110, w=250, h=220 | 敌人立绘、影子、少量特效 | 不承载关键文字 |
| 玩家表现区 | x=36, y=270, w=318, h=245 | 玩家立绘、内力光环、出招动势 | 不压住搓招面板和手牌 |
| 消息条 | x=24, y=462, w=246, h=28 | 简短状态反馈 | 只显示一句，不堆日志 |
| 出招按钮 | x=288, y=500, w=76, h=52 | 出招/禁用 | 与搓招面板同一行 |
| 搓招面板 | x=20, y=496, w=254, h=74 | 招式名、序列、预计伤害 | 背景深色，文字高对比 |
| 手牌区 | x=12, y=586, w=366, h=132 | 5-7 张碎片卡 | 卡牌必须可读，选中态上浮 |
| 底部玩家 HUD | x=16, y=726, w=210, h=34 | 玩家血量、护甲 | 不与卡牌重叠 |
| 底部操作区 | x=0, y=720, w=390, h=98 | 出招次数、清空、回合、内功简略 | 按钮不进入手势区 |

## 4. 资源尺寸规范

当前资源尺寸如下，绘制时必须尽量保持原始比例，不允许随意强行拉伸。

| 资源 | 当前尺寸 | 原始比例 | 推荐绘制规则 |
| --- | --- | --- | --- |
| `background.png` | 780 x 1688 | 0.462 | 可等比 cover 到 390 x 844，允许居中裁切 |
| `player.png` | 345 x 438 | 0.788 | 推荐显示宽 250-290，高度按比例计算 |
| `enemy.png` | 380 x 438 | 0.868 | 推荐显示宽 160-190，高度按比例计算 |
| `card-direction.png` | 204 x 255 | 0.800 | 推荐显示 54 x 68 或 60 x 75，不要画成窄高卡 |
| `card-punch.png` | 207 x 255 | 0.812 | 与方向卡统一外框尺寸 |
| `card-kick.png` | 202 x 255 | 0.792 | 与方向卡统一外框尺寸 |
| `panel.png` | 708 x 250 | 2.832 | 可用于宽面板，避免压到 63px 高导致细节糊 |
| `control-bar.png` | 1492 x 174 | 8.575 | 适合作为横向装饰条，不适合直接拉成高底栏 |

### 4.1 资源使用原则

- 背景必须降噪：战斗页背景上覆盖 35%-50% 暗色遮罩。
- 角色必须等比：玩家和敌人不得为了填满区域而变形。
- 卡牌必须清晰：卡面文字和图标优先于装饰纹理。
- 底栏资源只做装饰：关键数值和按钮仍由 Canvas 绘制，保证可读。
- 面板资源如果内部细节太复杂，优先使用 Canvas 深色面板承载文字。

## 5. 交互状态

### 5.1 未选牌

- 搓招面板显示“选择手牌碎片开始搓招”。
- 出招按钮禁用。
- 手牌全部正常高度。
- 消息条提示点击手牌组成序列。

### 5.2 只选方向

- 搓招面板显示方向序列。
- 出招按钮禁用。
- 提示“只有方向无法出招，需要加入拳或腿”。

### 5.3 未命中正式配方但含拳/腿

- 若序列含拳，显示“普通拳”。
- 若序列含腿，显示“普通腿”。
- 出招按钮可用。
- 预计伤害显示普通攻击伤害。

### 5.4 命中正式招式

- 搓招面板第一行显示招式名 + 输入序列。
- 第二行显示预计伤害和主要效果。
- 出招按钮高亮。
- 已选卡牌保持上浮或描边。

### 5.5 出招后

- 消耗已选卡牌。
- 清空搓招序列。
- 敌人血量立刻更新。
- 出招次数减少；若内功免耗，显示一次内功触发提示。

### 5.6 回合结束

- 手牌弃入弃牌堆。
- 敌人按意图行动。
- 玩家血量/护甲更新。
- 抽新手牌并进入下一回合。

## 6. 视觉规则

- HUD 永远在角色、背景和特效之上。
- 背景、角色、光环只负责氛围，不承载关键数值。
- 同一类信息使用同一位置：敌人状态只在顶部，玩家状态只在底部。
- 按钮使用稳定尺寸，点击区域不得小于 44 x 44。
- 文字不使用负字距，不跟随视口宽度缩放。
- 战斗页不放长篇说明，所有提示尽量控制在一行。
- 不使用微信胶囊右上区域显示敌人意图、护盾、金币等关键内容。

## 7. 后续实现验收标准

### 7.1 视觉验收

- 进入战斗不白屏。
- 敌人血量、敌人意图、玩家血量、手牌、搓招结果、出招按钮一眼可见。
- 微信右上胶囊不遮挡任何关键 HUD。
- 手牌不遮挡玩家血条。
- 卡牌、角色、底栏资源没有明显比例变形。
- 背景不会抢过手牌和搓招面板。

### 7.2 交互验收

- 点击手牌能加入序列。
- 点击已选碎片能撤回。
- 只选方向时不能出招。
- 有拳或腿但未命中正式配方时，可以出普通拳或普通腿。
- 命中正式配方时显示招式名和预计伤害。
- 出招、清空、回合按钮均可点击，且触摸区域与视觉位置一致。

### 7.3 自动化验收建议

后续实现时，浏览器预览版应补充：

- `window.render_game_to_text()`：返回当前场景、敌我血量、手牌、已选序列、当前招式、按钮可用状态。
- `window.advanceTime(ms)`：用于自动测试稳定推进渲染。

至少覆盖以下场景：

1. 标题页进入跑图。
2. 跑图进入战斗。
3. 战斗初始 HUD 可见。
4. 选择方向碎片后出招按钮禁用。
5. 选择拳/腿后普通攻击可用。
6. 命中配方后显示正式招式。
7. 出招后敌人血量变化。
8. 回合结束后敌人行动并抽新手牌。

## 8. 暂不处理内容

- 不在本阶段重做跑图、奖励、商店、事件、结算页面。
- 不在本阶段引入 Phaser、Cocos 或 Laya。
- 不在本阶段新增商业化、登录、排行榜、云存档。
- 不在本阶段继续批量换美术资源。
